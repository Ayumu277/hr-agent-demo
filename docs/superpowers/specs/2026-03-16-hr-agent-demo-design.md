# 人材業界向けAIエージェントデモ — 設計書

## 概要

人材派遣業の営業・コーディネーター向けに、AIエージェントが基幹システム（DB）・メール・カレンダー・ファイルサーバーを横断して業務を自動実行するデモアプリケーション。

**完全モック実装** — 外部サービスへの接続は一切不要。フロントエンドのみで動作する。

## アプローチ

**フロントエンドのみ（React + Vite + MUI）**

- バックエンド不要。`npm run dev` だけで起動
- シナリオ自動再生型（既存 aice-system-demo と同じ方式）
- ユーザーがチャットに入力 → 定義済みシナリオがタイマーで順次再生
- ツール実行結果はすべてハードコードされたモックデータ

## 画面構成

### 1. ダッシュボード (`/`)

モックデータの一覧表示。エージェントが操作する「基幹システム」のイメージを伝える画面。

**表示内容:**

- **スタッフ一覧テーブル**: 名前、スキル（タグ）、稼働状況（稼働中/待機中/契約終了間近）、現在の派遣先
- **案件一覧テーブル**: クライアント名、勤務地、必要人数、充足率（プログレスバー）、ステータス
- **契約一覧テーブル**: スタッフ名、クライアント名、契約期間、ステータス（稼働中/更新待ち/期限切れ間近）

各テーブルは MUI DataGrid を使用。ヘッダーに件数とステータス別のサマリーチップを表示。

### 2. AIエージェント (`/agent`)

チャットUI + シナリオ自動再生。メイン画面。

**構成要素:**

- **おすすめチップ**: 画面上部に5つの業務シナリオ候補を表示。クリックでシナリオ発動
- **メッセージエリア**: ユーザーメッセージ / アシスタントメッセージ / ツール実行を時系列で表示
- **入力エリア**: テキスト入力 + 送信ボタン。再生中は無効化
- **リセットボタン**: チャット履歴をクリア

### ナビゲーション

左サイドバーで2画面を切り替え:
- ダッシュボード（Dashboard アイコン）
- AIエージェント（SmartToy アイコン）

## シナリオ設計

### 共通仕様

- 各シナリオは `ScenarioStep[]` 配列として定義
- ステップの種類: `user` / `assistant` / `tool_call`
- `tool_call` は `running` → `completed` の2段階でアニメーション表示
- **各シナリオは独立したスナップショット** — シナリオ間でデータの整合性は不要。各シナリオが独自の世界観を持つ
- ツール種別はツール名のプレフィックスから自動判定:
  - `db_*`: `Storage` アイコン（青）
  - `email_*`: `Email` アイコン（緑）
  - `calendar_*`: `CalendarMonth` アイコン（オレンジ）
  - `file_*`: `FolderOpen` アイコン（紫）
- **タイミング**: assistant メッセージ 1000ms、tool_call の running 表示 1500ms、completed 遷移 1000ms（既存デモと同等）
- **チップクリック動作**: クリック時にトリガーテキストを入力欄に挿入して自動送信（ユーザーが入力したように見せる）
- **自動スクロール**: メッセージ追加時にチャットエリアを最下部にスクロール

### シナリオ1: 案件充足の自動化

**トリガー**: 「来週充足率が低い案件を調べて、マッチするスタッフをアサインして」

| ステップ | 種別 | 内容 |
|---------|------|------|
| 1 | assistant | 「承知しました。来週の案件と充足率を確認します。」 |
| 2 | tool_call | `db_search_jobs` → 「来週の案件15件を取得。充足率50%未満が3件」 |
| 3 | assistant | 「充足率が低い3件を特定しました。条件にマッチするスタッフを検索します。」 |
| 4 | tool_call | `db_search_staff` → 「条件に合う待機中スタッフ6名を抽出」 |
| 5 | tool_call | `db_assign_staff` → 「3件の案件に4名をアサイン」 |
| 6 | assistant | アサイン結果の詳細（案件名・スタッフ名・充足率変化）を表示 |

**モックデータ（結果表示）:**
```
■ 物流センターA 仕分け作業（千葉県船橋市）
  → 田中太郎をアサイン｜充足率 50% → 100%

■ コールセンター 受電業務（東京都新宿区）
  → 佐藤花子、鈴木一郎をアサイン｜充足率 33% → 100%

■ イベント会場 設営補助（神奈川県横浜市）
  → 高橋美咲をアサイン｜充足率 0% → 50%
```

## シナリオ選択ロジック

チャット入力テキストに含まれるキーワードでシナリオを判定。**上から順に評価し、最初にマッチしたシナリオを実行**:

| 優先順 | シナリオ | マッチキーワード |
|-------|---------|----------------|
| 1 | 案件充足 | `充足`, `アサイン`, `マッチ` |
| 2 | フォローアップ | `フォロー`, `契約切れ`, `契約終了`, `期限切れ`, `次の案件` |
| 3 | クライアント対応 | `依頼`, `提案書`, `新着メール` |
| 4 | 勤怠異常 | `勤怠`, `異常`, `出勤`, `欠勤` |
| 5 | 契約更新 | `契約更新`, `更新手続き`, `更新書類` |

マッチしない場合は「申し訳ありません。以下のシナリオから選択してください」と5つのチップを再表示。

**キーワード設計の注意**: `期限` は曖昧なためシナリオ2に `期限切れ` として含め、シナリオ5は `契約更新` `更新手続き` `更新書類` でマッチ。`メール` `候補` `案件` など汎用的すぎるワードは除外。

## モックデータ構造

フロントエンド内に `data/` ディレクトリを作成し、JSONとして保持。
**ダッシュボード表示用のデータのみ**。勤怠・メール・カレンダーのデータはシナリオ定義内のハードコード文字列として存在し、別途JSONファイルは不要。

`contracts.json` にはシナリオ2用（3月末期限）とシナリオ5用（4月末期限）の両方の契約を含める:

### staffs.json
```typescript
interface Staff {
  id: string;
  name: string;
  skills: string[];
  status: 'working' | 'available' | 'expiring';
  currentClient?: string;
  qualifications: string[];
}
```

### jobs.json
```typescript
interface Job {
  id: string;
  clientName: string;
  location: string;
  requiredCount: number;
  assignedCount: number;
  skills: string[];
  hourlyRate: number;
  status: 'open' | 'partially_filled' | 'filled';
}
```

### contracts.json
```typescript
interface Contract {
  id: string;
  staffId: string;
  staffName: string;
  clientName: string;
  startDate: string;
  endDate: string;
  status: 'active' | 'renewal_pending' | 'expiring_soon';
}
```
